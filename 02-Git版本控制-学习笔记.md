## 1. 什么是 Git（大白话）

**Git 就是给你的文件加了一套"游戏存档"。**

- 打游戏你会**存档**：开头存一个、打到一半存一个、最新再存一个。万一打死了、走错路了，就**读档**退回上一个存档点。
- Git 干的就是这件事，只不过它存的不是游戏进度，是**你的文件**。
- 而且它比游戏存档更聪明：它记得你每次存档**改了哪几行、什么时候改的、为什么改**。

**那 GitHub 是什么？** 它是把这些存档**放到网上的地方**——既是云备份（电脑坏了不怕），也是个分享广场，全世界的人把自己的项目（存档）放上去。你学习用的那个学习地图，就是别人放在 GitHub 上的一份项目。

**为什么必须学：** 以后你用 AI 写代码，AI 一下子改一堆文件，git 能让你看清"它到底改了啥"、改坏了**一键退回**——这是用 AI 写代码的**安全绳**。

## 2. 开始前的准备

### 确认 git 装好了

```
git --version
```

- 显示类似 `git version 2.54.0` → 已装好。
- 显示红字"无法将 git 识别为命令" → 还没装，需要先去 git 官网下载安装。

### 做一次性的"自我介绍"

git 每次存档都会记"这是谁存的"，所以要先告诉它你的名字和邮箱。**一台电脑只设一次，终身通用。**

```
git config --global user.name "Liz"
git config --global user.email "tank2024001@gmail.com"
```

- `config` = 设置，`--global` = 全局通用。
- 回车后没反应是正常的（默默记下了）。

**验证记住没有**（反过来问它，不带引号内容）：

```
git config --global user.name
git config --global user.email
```

应该分别回显你填的名字和邮箱。

> ⚠️ **务必一行一条、各自回车！** 两条命令挤在一行会报 `no action specified`（没说清要干啥）。从聊天里复制时，**一行一行复制粘贴**，别一次粘两行。

## 3. 核心流程总览

git 最日常的一条流程，你以后会用一辈子：

```
git init   →   改文件   →   git status   →   git add   →   git commit -m   →   git log
（开启存档）   （干活）      （看状态）       （装箱）        （正式存档）         （翻历史）
```

下面逐个拆开讲。

---

## 4. git init —— 开启存档功能

- **大白话：** 让 git 开始盯着、管理这个文件夹。相当于"装监控、开机"。
- **来历：** **init**ialize（初始化）。
- **前提：** 先 `cd` 进入你要管理的项目文件夹（所有 git 操作都得站在项目里做）。
- **操作：**
  ```
  cd ~
  cd 练习
  git init
  ```
  回显类似：`Initialized empty Git repository in C:/Users/40315/练习/.git/`

- **背后发生了什么：** 它在文件夹里偷偷建了一个隐藏的 `.git` 文件夹——这就是 **存档库（仓库 / repository）**，以后所有存档和历史都装在这里面。从此这个文件夹就升级成了一个 **git 项目**。
- **确认它建好了：**
  ```
  ls -Force
  ```
  能看到一个 `.git`（显示 `d--h--`，d=文件夹、h=隐藏），就成功了。

---

## 5. git status —— 看现在啥情况

**这是你以后最常用的命令，没有之一。** 它汇报"现在啥情况"。读懂它，就掌握了一半 git。

**两个关键状态 + 颜色：**

| 状态 | 颜色 | 含义 |
|---|---|---|
| `Untracked files`（未跟踪） | 红 | 这个文件 git 看得见，但**还没纳入管理**（新文件第一次出现） |
| `Changes not staged` → `modified`（已修改） | 红 | 一个**已被管理**的文件，你改动了，但还没装箱 |
| `Changes to be committed` → `modified` / `new file` | 绿 | 文件已经**装进箱（add 过）**，等着封存 |
| `nothing to commit, working tree clean` | —— | 干净！没有任何待处理的改动 |

**记住颜色规律：红色 = 还没进箱；绿色 = 已进箱待存。**

> 小贴士：`git status` 里那些括号灰字（如 `use "git add..."`）是 git 在**手把手教你下一步**，别划过去，经常直接给你答案。

## 6. git add + commit —— 存档两步走

存档像**寄快递，分两步**：

### 第一步：`git add` —— 装箱

- **大白话：** 挑出"这次存档要包含哪些文件"，放进待发的箱子。
- **操作：**
  ```
  git add note.txt
  ```
  回车没反应，正常。之后 `git status` 看，文件会从**红变绿**。
- **为什么要单独这一步？** 因为有时你改了一堆文件，但只想存其中几个，`add` 就是让你挑。

### 第二步：`git commit -m "说明"` —— 封箱寄出

- **大白话：** 正式存档，并贴一张标签说明这次干了啥。
- **拆解：** `commit` = 正式存档；`-m` = message（留一句说明）；引号里 = 你的**提交信息（标签）**。
- **操作：**
  ```
  git commit -m "我的第一次存档"
  ```
  回显一张"小票"，类似：
  ```
  [master (root-commit) a1138fb] 我的第一次存档
   1 file changed, 1 insertion(+)
  ```
- **小票怎么读：** `root-commit` = 这是项目第一次存档；`a1138fb` = 这次存档独一无二的**编号（身份证号）**；`1 file changed` = 改了几个文件；`insertion(+)` = 新增了几行。

> ⚠️ **`commit` 一定要带 `-m "说明"`！** 漏了会掉进一个叫 Vim 的编辑器（见第 11 节逃生）。

> 💡 **标签要写具体：** 练手写"我的第一次存档"没问题，真实项目里大家会写 `添加登录功能`、`修复xx错误` 这种，方便未来的自己一眼认出。

## 7. git log —— 翻看历史

- **大白话：** 把你存过的所有档列出来，看谁存的、啥时候、存了啥说明。
- **操作：**
  ```
  git log              # 完整版
  git log --oneline    # 每条压成一行，清爽版（推荐）
  ```
- **`git log` 怎么读：**
  - `commit a1138fbc53b...` —— 完整编号（小票上的 7 位是它的简写）
  - `(HEAD -> master)` —— **HEAD = 你现在站在哪个存档上**。它跟着哪条，你就在哪条。"读档"穿越历史，本质就是移动这个 HEAD。
  - `Author: Liz <...>` —— 谁存的（你最早设的自我介绍在这派上用场）
  - `Date: ...` —— 什么时候存的
  - 最下面那行 —— 你写的标签（为什么/存了啥）
- **顺序：** 从上到下 = 从新到旧（像刷朋友圈）。

> 💡 **Vim 逃生预备：** `git log` 历史很长时会进入**只能看、不能打字**的浏览模式（底部出现 `:`）。按一下 **`q`** 键即可退出。

## 8. 完整走一遍：第一次 + 第二次存档

**第一次存档**（针对一个新文件）走完后，体会不到 git 的妙处。**改一下文件、存第二次**，"历史"的味道才出来：

```
echo "这是我添加的第二行" >> note.txt   # 用 >> 追加（保留原内容）
cat note.txt                            # 确认变成两行
git status                              # 这次报 modified（已修改，红色）
git add note.txt                        # 装箱（变绿）
git commit -m "添加了第二行内容"          # 封存，标签写具体些
git log --oneline                       # 看到两条记录一上一下排着 = 历史诞生了
```

> 📌 **关于 `0 insertions` 的小知识（Windows 专属坑）**
> 用单个 `>` 在 PowerShell 造的文件，末尾没留正经的"换行符"，导致后面 `>>` 追加时，git 可能把它算成"把原来那行改长了"而不是"新增一行"，于是显示 `0 insertions`。**这不影响存档成功**，纯属换行符（CRLF/LF）的小问题。以后用 VS Code 这类正经编辑器写文件就不会有。现在完全不用深究。

## 9. 后悔药 ①：git restore（没commit时）

- **大白话：** 把文件**还原成上一次存档时的样子**，刚才的改动不要了。
- **适用：** 改动**还没 commit**（红色没进箱、或绿色已进箱，都能救）。
- **两种情况：**

**情况 A —— 改了但还没 add（红色）：** 一步还原内容
```
git restore note.txt
```

**情况 B —— 已经 add 进箱了（绿色）：** 分两步
```
git restore --staged note.txt   # 第一步：先把它退出暂存箱（内容先不动，绿→红）
git restore note.txt            # 第二步：再把内容还原成上次存档的样子
```

- 还原后 `cat note.txt`，你会看到乱改的内容**凭空消失**，文件变回干净的样子。
- **比喻：** restore 是"东西还没交上去（没封箱），随手把草稿一撕"。

## 10. 后悔药 ②：git revert（已commit后）

**关键分界：`commit` 是一道"封箱线"。** 一旦垃圾内容已经 `commit -m` 进了历史，`git restore` 就救不了了，要换 `git revert`。

- **大白话：** 它**不撕历史**，而是**新建一条存档**，效果正好把那次的改动**抵消掉**。
- **怎么用（必须指定撤销哪一条，靠编号）：**
  ```
  git log --oneline          # 先找到要撤销那条的编号，比如 4f880b0
  git revert 4f880b0         # 撤销它
  ```
- **⚠️ 会弹编辑器：** revert 默认弹出 Vim 让你确认说明，但它**已自动填好**，你**什么都不用改，保存退出**即可——按 `Esc`，输入 `:wq`，回车（见第 11 节）。
- **结果（看起来矛盾、其实精妙）：**
  ```
  git log --oneline   # 历史不减反增，多了一条 "Revert ..."；垃圾那条还留着
  cat note.txt        # 文件却变干净了
  ```

> **为什么历史还留着垃圾、文件却干净了？——"会计记账"比喻**
> 想象一本不能擦写的账本：
> - 垃圾那次 = 记了一笔「+ 一行垃圾」
> - revert = 再记一笔「− 一行垃圾」
> - 一加一减，**账面余额（你的文件）回到出错前**，但两笔记录都白纸黑字留着。
>
> **这么设计的三大好处：**
> 1. **历史诚实可追溯** —— 看得到"我犯过错、又在哪步修好"。
> 2. **绝对安全** —— revert 只新增、从不删除，不会弄丢东西、不破坏别人的历史。
> 3. **可逆的可逆** —— 连撤销本身都能再撤销。

> ⚠️ 你可能听过 `git reset`，它是真能"撕掉"历史的（更生猛），但有风险、多人协作会闯祸。**现阶段只用 `revert` 就够了**，`reset` 以后真有需要再学。

**一句话分界：** restore 是"记账前划掉草稿"，revert 是"已记账就再记一笔反向账"。分界线就是 **commit**——封箱前用 `restore`，封箱后用 `revert`。

## 11. Vim 逃生指南

掉进 Vim（满屏、底部一堆 `~`、打字没反应）别慌，记住这对口诀：

| 你想干嘛 | 操作 |
|---|---|
| **不保存、丢弃退出**（比如 commit 漏了 -m 想取消） | 按 `Esc` → 输入 `:q!` → 回车 |
| **保存退出**（比如 revert 时保留它自动写好的说明） | 按 `Esc` → 输入 `:wq` → 回车 |

- 打 `:` 冒号要按 **Shift + 分号**。
- 打错了多按几次 `Esc` 重来，怎么都弄不坏东西。
- 助记：**`q!` = quit 丢弃；`wq` = write 写入保存 + quit。**

## 12. 踩过的坑（很值钱）

这些是真实世界里几乎人人会踩的坑，你已经亲身爬出来过，比看十遍教程都记得牢：

| 坑 | 现象 | 自救 |
|---|---|---|
| 两条命令写在同一行 | `no action specified` | 一行一条、各自回车；别一次粘两行 |
| `commit` 漏了 `-m` | 掉进 Vim 满屏 `~` | `Esc` → `:q!` → 回车 |
| `revert` 弹出编辑器 | 又掉进 Vim | `Esc` → `:wq` → 回车（保留自动说明） |
| 文件名打错 / 走错文件夹 | `PathNotFound` 红字 | `pwd` 看位置、`ls` 看文件、名字打对重来 |
| `revert` 没带编号 | git 把一长串用法说明摊给你 | 加上 `git log --oneline` 里那条的编号 |

## 13. 命令总结表（小抄）

| 命令 | 干啥的 | 比喻 |
|---|---|---|
| `git --version` | 看 git 装没装、哪个版本 | 验明正身 |
| `git config --global user.name/email` | 一次性自我介绍 | 登记存档人 |
| `git init` | 让 git 开始管这个文件夹 | 装监控、开机 |
| `git status` | 看现在啥情况（**最常用**） | 查看现场 |
| `git add 文件` | 把文件放进待存档的箱子 | 装箱 |
| `git commit -m "说明"` | 正式存档 + 贴标签 | 封箱寄出 |
| `git log` / `git log --oneline` | 回看所有历史存档 | 翻档案 |
| `git restore 文件` | 丢弃没commit的改动 | 撕掉草稿（后悔药①） |
| `git restore --staged 文件` | 把文件退出暂存箱 | 从箱里拿出来 |
| `git revert 编号` | 撤销已commit的存档 | 记一笔反向账（后悔药②） |

**一句话总结：** `init` 开监控 → 改文件 → `status` 看状态 → `add` 装箱 → `commit` 封存 → `log` 翻历史。改坏了别怕：没封箱用 `restore`，已封箱用 `revert`。

## 14. 还没学 / 下一步

这份笔记覆盖了我们实操过的 git **本地**核心。以下是**还没学、但很快会用到**的，先记个名字：

- **`git clone 网址`** —— 把别人 GitHub 上的项目**整个下载**到你电脑（你学习地图的下一步就靠它）。
- **`git push` / `git pull` + GitHub 账号** —— git 的**"上传到云 / 从云下载"**这半边。Stage 0 要求会 `push`，需要先注册 GitHub 账号、把本地项目和网上的仓库连起来。
- **`.gitignore`** —— 告诉 git "这些文件别管"（比如临时文件、密码文件）。真实项目几乎都有。
- **`git diff`** —— 看"这次到底改了哪几行"的精确对比。
- **分支 branch** —— 我们见过 `master` 这个词，但还没真正用。以后做大项目、多人协作会学到。
- **`git reset`** —— 更生猛的"撕历史"工具，进阶再说。

> 这些等你想推进 Stage 0 时随时找我，照样一步一步带你过。
